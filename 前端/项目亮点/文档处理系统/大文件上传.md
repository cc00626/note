## 1.背景

在上传压缩包的过程中，因为我们上传压缩包普遍大小比较大，通常都在200MB--300MB左右，在上传的过程中，发现需要等待很长时间才能显示，并且一但进行刷新，就需要重新进行上传。所以我针对这个问题进行主要的优化。

## 2.技术调研

我们针对大文件需要分成小块，同时还要保证后端接受到小块之后的数据完整性。首先是和后端确定好，每一片的大小，我们确定的是每一片为5M，然后为了保证数据的完整性会使用md5计算出文件的哈希值，然后确定每一个文件的哈希值为完整文件的哈希值带上index，为了保证文件合并的时候顺序。



**断点续传** ：在每一次发送前先发送一个请求。

后端根据文件的hash值找到对应的分片然后将索引值返回前端，前端通过数组的filter方法进行过滤找到没有上传的文件。



**保证文件的完整性**：在合并的时候会将整个文件的哈希值发送给后端。后端合并完整之后对文件再次进行md5加密，保证文件的完整性。如果不是完整的会给前端一个错误提示。让前端重新进行上传。



**保证文件的顺序**：在上传文件的时候会带上一个chunkhas这个值是通过整片文件的hash值加上一个文件的index。后端合并的时候需要对文件名进行一个拆分，然后根据索引进行排序来进行合并。



我们设计下面几个接口

**发送接口**  /upload   

参数   `chunk`（切片文件）、`chunkHash`（切片 hash是文件的哈希值带上index）、`fileName`、`fileHash`

## 3.用户体验优化

### 控制并发上传

浏览器对同一个域名的并发请求有限制 一般是6到8个，不过不进行控制，请求就会排队，反而会是上传的速度降低。

```typescript
  let index = 0
  const max = 6
  const taskPool = []
  while (index < formDatas.length) {
    while (taskPool.length < max && index < formDatas.length) {
      const task = fetch('http://localhost:3000/upload', {
        method: 'POST',
        body: formDatas[index]
      }).then(() => {
        taskPool.splice(taskPool.indexOf(task), 1)
      })
      taskPool.push(task)
      index++
    }
    await Promise.race(taskPool)
  }

  await Promise.all(taskPool)
```

### 为什么使用MD5加密？

- **计算速度快**：MD5 是一种快速的哈希算法，在文件上传场景中，它能够在较短的时间内计算出文件的哈希值。尽管相较于 SHA-256，MD5 的哈希值更短（128 位），但它对于大文件的处理速度具有明显优势，适合需要较高性能的场景。
- **广泛的支持与兼容性**：MD5 在各大平台和库中得到广泛支持，能够轻松地与前后端进行集成。
- **合适的准确性与足够的安全性**：尽管 MD5 存在碰撞问题（即不同的输入可能会产生相同的哈希值），但在大文件上传的场景下，由于文件大小和结构的独特性，发生碰撞的几率极低，足以保证文件的完整性。如果对安全性有更高要求，可以考虑使用更复杂的算法（如 SHA-256），但在大文件上传的效率要求下，MD5 已经足够应对大部分场景。
