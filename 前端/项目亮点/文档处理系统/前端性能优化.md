## 1.背景

老师反映项目首页加载太慢，因为需要给甲方进行演示，希望演示更加流畅。首页从开始到显示有一定的白屏时间。通过performce检查发现FCP普遍在3s左右。

## 2.怎么做

通过浏览器的performance面板进行分析，发现部分都发生在了js加载和解析的时候，于是我就针对js代码的体积进行了一系列优化。



**将lodash等Commonjs模块的包改为ESM模块的包，为了更好的支持webpack的tree skating**在生产模式下会自动开启webpack的treeskating。

如果是在开发模式下，需要将webpack的useexport设置true，packjson中这是sideEffect设置成true，表示文件产生了副作用。不要进行删除。如何一些文件进行了全局的修改，但是并没有导出函数，webpack就不知道要不要进行删除，一般情况下设置css文件。



**使用懒加载，将文件的第三方库分离。**

```typescript
  optimization: {
    splitChunks: {
      chunks: 'all', // 对所有模块生效（异步和非异步）
      minSize: 20000, // 模块超过 20kb 才提取
      maxSize: 0, // 不限制最大体积
      minChunks: 1, // 至少被引用 1 次
      cacheGroups: {
        // 提取第三方库
        vendors: {
          test: /[\\/]node_modules[\\/]/,
          name: 'vendors',
          priority: -10,
          reuseExistingChunk: true,
        },
        // 提取公共代码
        common: {
          name: 'common',
          minChunks: 2, // 被至少两个模块引用
          priority: -20,
          reuseExistingChunk: true,
        },
      },
    },
```



**使用代码压缩，减少文件的体积。**

使用webpack-treser插件 压缩代码的体积。

```typescript
  optimization: {
    minimize: true, // 启用压缩
    minimizer: [
      new TerserPlugin({
        parallel: true, // 开启多进程压缩，提速
        extractComments: false, // 不生成 LICENSE.txt
        terserOptions: {
          compress: {
            drop_console: true, // 删除 console.log
            drop_debugger: true, // 删除 debugger
            pure_funcs: ['console.info'], // 额外删除指定函数
          },
          mangle: true, // 混淆变量名
          format: {
            comments: false, // 移除注释
          },
        },
      }),
    ],
  },
```



**开启gzip进行代码压缩**

```typescript
server {
    listen 80;
    server_name yourdomain.com;
    root /var/www/your_project/dist;
    index index.html;

    # SPA 路由
    location / {
        try_files $uri /index.html;
    }

    # 静态资源缓存
    location ~* \.(js|css|png|jpg|jpeg|gif|svg|ico)$ {
        expires 1y;
        add_header Cache-Control "public";
    }

    # HTML 不缓存
    location /index.html {
        expires -1;
        add_header Cache-Control "no-cache";
    }

    # gzip 压缩
    gzip on;
    gzip_min_length 1024;
    gzip_types text/plain application/javascript text/css application/json;
    gzip_vary on;

    # 反向代理（可选）
    location /api/ {
        proxy_pass http://backend-server:3000/;
        proxy_set_header Host $host;
    }
}
```

## 3.结果

通过webpack-bunle-analyze 分析代码体积，查看分包是否完成。配合后面的数据预加载和流式html  将FCP从2.9s下降到1.8s。



## 如何进行性能优化

“在最近的 WebGIS 毕业项目中，老师反馈首页白屏时间过长，影响给甲方的演示体验。我使用 Chrome DevTools 的 **Performance** 面板进行了分析，发现 **FCP（首次内容绘制）** 普遍在 **3s** 左右。通过性能图表观察到，主线程在加载初期有长时间的 **灰色长任务**，且大部分耗时集中在脚本的 **评估（Evaluate）** 和 **执行（Execution）** 上，这说明 JS 文件的体积过大是核心瓶颈。”



**开启 Tree Shaking：** 我将像 `lodash` 这样的 CommonJS 库替换为支持 **ESM** 的版本，并在 Webpack 中配置了 `usedExports`。同时在 `package.json` 中合理设置 `sideEffects`（例如保留 CSS 文件的副作用），确保未使用的冗余代码被彻底剔除。



**精细化分包（SplitChunks）：** 我利用 Webpack 的 `splitChunks` 插件，将 `node_modules` 中的第三方重型库（如 AMap、Echarts）抽离到 `vendors` 包中，并结合 **路由懒加载**，确保首屏只加载必需的代码。



**极致压缩：** 使用 `TerserPlugin` 开启多进程并行压缩，并在生产环境下剔除了所有 `console.log` 和注释，进一步减小源码体积。

因此我针对js体积进行了优化。



“最后，我通过 `webpack-bundle-analyzer` 验证了分包逻辑的准确性。配合后续的数据预加载优化，**FCP 从 2.9s 下降到了 1.8s**，提升了约 **38%** 的加载速度。白屏感明显消失，顺利完成了演示任务。”



## 依据什么进行分包

我分包的核心逻辑是**将“几乎不动”的第三方库与“经常变动”的业务代码解耦**，从而最大化利用浏览器的缓存机制。

####  1. 依据第三方库的稳定性（Vendors 分包）

- **原则：** 把 `node_modules` 里的第三方依赖（如 React, Vue, Lodash, Echarts）单独打包。
- **理由：** 这些库的版本通常不会变动。单独打成一个 `vendors.js` 后，只要版本不变，文件名（Hash）就不会变。用户下次访问时，浏览器可以直接从 **Disk Cache** 读取，不需要重新下载几百 KB 的代码。

#### 2. 依据引用频率（Common 分包）

- **原则：** 提取被多个页面/路由共同引用的公共模块（比如自定义的组件、工具函数 Utils）。
- **理由：** 避免同一个组件在 A 页面包里有一份，B 页面包里也有一份，造成冗余下载。通过设置 `minChunks: 2`，可以把这些公共逻辑提取到 `common.js` 中。

#### 3. 依据“当前页面必需性”（路由懒加载/异步分包）

- **原则：** 使用 `import()` 动态导入，配合 `chunks: 'async'`。
- **理由：** **首屏没用到的代码，绝对不让用户在首屏下载。** 比如 WebGIS 项目里的“后台管理页”或“导出统计表”功能，用户不点击，对应的 JS 就永远不加载。
