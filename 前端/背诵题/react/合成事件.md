### 什么是react的事件委托

在原生 DOM 开发中，给每个按钮绑定事件很浪费内存。React 采用了一种更聪明的做法：**它不会把事件处理器直接挂在真实的 DOM 节点上**，然后通过事件冒泡机制，触发根节点的事件。

- **统一监听：** React 会在特定的节点上通过 `addEventListener` 监听所有的原生事件。
- **版本差异：**
  - **React 16 及之前：** 事件被委托到了 `document` 上。
  - **React 17/18 及之后：** 事件被委托到了 **渲染树的根节点（Root Container）** 上。



### 什么是react合成事件

React 合成事件（SyntheticEvent）是 React 对浏览器原生事件的一层封装，目的是实现跨浏览器兼容、统一事件行为，并提升性能。

在实现上，React 采用了**事件委托机制**，不会给每个 DOM 节点单独绑定事件，而是把事件统一绑定在根节点上。当事件触发时，React 会先接收到原生事件，然后构造一个合成事件对象，再分发给对应的组件处理函数。

合成事件的 API 和原生事件基本一致，比如 `preventDefault`、`stopPropagation` 等，但内部做了兼容处理。

它的优势主要有三点：

第一，**跨浏览器统一**。开发者不需要关心不同浏览器的事件差异。

第二，**性能优化**。通过事件委托，减少大量事件监听器的创建。

第三，**可控性更强**。React 可以控制事件的执行时机，比如在事件中进行状态更新时，会进行批量更新（batch update），提高性能。

需要注意的是：

- 在 React 17 之前，合成事件有事件池机制（event pooling），事件对象会被复用，异步访问时需要调用 `e.persist()`。
- React 17 之后已经取消了事件池。
- 合成事件和原生事件混用时，可能会出现执行顺序问题。

### 为什么 React 17 要改变委托对象？（必考题）

面试官最喜欢问：**“为什么不往 `document` 上挂了？”** * **多版本共存：** 如果一个页面上有两个不同版本的 React，它们都把事件挂在 `document` 上，事件冒泡就会乱套。改到根节点后，各个 React 应用的事件互不干扰。

- **微前端支持：** 在微前端架构中，子应用通常挂在某个 DOM 下。改到根节点能让事件作用域更安全，避免影响到宿主环境或其他框架。



React 并不是直接传递原生 `Event` 对象给你，而是包装了一个 **`SyntheticEvent`**。

- **抹平差异：** 不同浏览器（Chrome、Safari、IE）的事件表现不一。合成事件提供了一致的接口，实现“一处编写，到处运行”。
- **垃圾回收与池化：**
  - 在 React 16 中，为了性能，合成事件会放入一个“对象池”复用。如果你在异步操作（如 `setTimeout`）里访问 `event`，它可能已经被置空了（需要调用 `event.persist()`）。
  - **在 React 17/18 中：** 性能已经足够快，**事件池化（Event Pooling）机制已被删除**，你现在可以直接在异步函数里用事件对象了。



当用户点击一个按钮时，React 内部经历了什么？

1. **原生冒泡：** 事件触发，一路冒泡到 Root 节点。
2. **合成事件提取：** React 的事件监听器截获事件，根据触发的 DOM 节点找到对应的 **Fiber 节点**。
3. **收集处理器：** React 会从当前的 Fiber 节点开始，向上遍历 Fiber 树，收集所有的事件回调（比如 `onClick`, `onClickCapture`）。
4. **分发执行（Dispatch）：**
   - 先执行**捕获阶段**的回调。
   - 再执行**冒泡阶段**的回调。
5. **触发更新：** 在回调执行过程中，你可能会调用 `setState`，这会进入我们之前聊到的 **Render 阶段**，开始构建 `workInProgress` 树并打上 `effectTag`。



### 事件系统中的批量更新

React 的事件系统是基于合成事件实现的，它通过统一接管事件触发入口，使得 React 可以在事件回调中对状态更新进行集中调度，从而实现**批量更新（batching）**，减少不必要的多次渲染，提高性能。



React 使用了一种类似于“状态开关”的机制：

1. **触发事件**：用户点击按钮，浏览器触发原生事件。
2. **进入合成事件层**：React 捕获到事件，找到对应的 `onClick` 回调。
3. **开启批量更新开关**：React 内部设置 `isBatchingUpdates = true`。
4. **执行回调函数**：
   - 执行 `setCount(c => c + 1)` -> React 发现开关是开的，将更新存入队列，**不渲染**。
   - 执行 `setFlag(f => !f)` -> 同样存入队列，**不渲染**。
5. **函数执行完毕**：回到合成事件层。
6. **关闭开关并生效**：设置 `isBatchingUpdates = false`，此时 React 才会触发**唯一一次** Virtual DOM Diff 和页面重绘。



| **特性**                     | **React 17 (及以前)**            | **React 18+**          |
| ---------------------------- | -------------------------------- | ---------------------- |
| **合成事件中**               | ✅ 批量更新                       | ✅ 批量更新             |
| **setTimeout / setInterval** | ❌ 独立更新（多次渲染）           | ✅ **批量更新**         |
| **Promise / await**          | ❌ 独立更新（多次渲染）           | ✅ **批量更新**         |
| **原生 DOM 事件**            | ❌ 独立更新（多次渲染）           | ✅ **批量更新**         |
| **底层依赖**                 | 同步标志位 (`isBatchingUpdates`) | 优先级调度与微任务队列 |



### react的事件和普通的html事件有什么区别

区别：

- 对于事件名称命名方式，原生事件为全小写，react 事件采用小驼峰；
- 对于事件函数处理语法，原生事件为字符串，react 事件为函数；
- react 事件不能采用 return false 的方式来阻止浏览器的默认行为，而必须要地明确地调用`preventDefault()`来阻止默认行为。



合成事件是 react 模拟原生 DOM 事件所有能力的一个事件对象，其优点如下：

- 兼容所有浏览器，更好的跨平台；
- 将事件统一存放在一个数组，避免频繁的新增与删除（垃圾回收）。
- 方便 react 统一管理和事务机制。



当你点击组件中的一个按钮时，事件的物理路径如下：

1. **原生捕获阶段 (Native Capture)**： 事件从 `window` 向下经过 `document`，到达 React 的挂载点 `Root`。
2. **React 捕获阶段 (Synthetic Capture)**： 当原生事件到达 `Root` 时，React 立即接管。它在内存中建立一条从 `Root` 到 `Target`（目标按钮）的路径，依次触发你写的 `onClickCapture` 函数。
3. **原生目标阶段 (Native Target)**： 事件到达真正的 DOM 节点。如果你在按钮上通过 `addEventListener` 绑定了原生事件，它会在这里执行。
4. **React 冒泡阶段 (Synthetic Bubble)**： React 在内存中模拟从 `Target` 回到 `Root` 的路径，依次触发 `onClick` 函数。
5. **原生冒泡阶段 (Native Bubble)**： 原生事件离开 `Root`，继续向上传递给 `document` 和 `window`
