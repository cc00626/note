

## requestAnimationFrame

requestAnimationFrame最大的优势是由系统来决定回调函数的执行时机。具体一点讲，如果屏幕刷新 率是60Hz,那么回调函数就在16.7ms（60Hz）后执行一次，如果刷新率是75Hz，那么这个时间间隔就变 成了1000/75=13.3ms，换句话说就是，「requestAnimationFrame的步伐跟着系统的刷新步伐 走」，「它能保证回调函数在屏幕每一次的刷新间隔中只被执行一次」，「这样就不会引起丢帧现象， 也不会导致动画出现卡顿的问题」



这是区分普通开发者和高级开发者的关键。**rAF 既不是微任务，也不是宏任务。**

在事件循环中，它属于**渲染阶段 (Rendering Pipeline)** 的一部分。一轮完整的循环如下：

1. **执行宏任务** (Task)
2. **清空所有微任务** (Microtasks)
3. **渲染更新** (Update the rendering):
   - 触发 `resize` / `scroll` 事件
   - **执行 `requestAnimationFrame` 回调**
   - 执行 `IntersectionObserver`
   - **计算样式与布局 (Layout/Reflow)**
   - **绘制 (Paint)**

## cookie和storage

| **特性**       | **Cookie**                             | **LocalStorage**                | **SessionStorage**           |
| -------------- | -------------------------------------- | ------------------------------- | ---------------------------- |
| **生命周期**   | 可设置过期时间，否则默认关闭浏览器失效 | **永久保存**，除非手动删除      | **仅在当前会话（窗口）**有效 |
| **存储大小**   | 非常小（约 **4KB**）                   | 较大（约 **5MB**）              | 较大（约 **5MB**）           |
| **服务端通信** | **自动携带**在 HTTP 请求头中           | 仅在客户端保存                  | 仅在客户端保存               |
| **数据操作**   | 比较麻烦（需自行封装）                 | 原生 API（`setItem`/`getItem`） | 原生 API                     |

**`HttpOnly`**：**最重要**。设置后，JS 无法通过 `document.cookie` 读取。**能有效防止 XSS 攻击盗取 Cookie。**

**`Secure`**：只在 HTTPS 协议下传输。

**`SameSite`**：

- `Strict`：完全禁止第三方 Cookie。
- `Lax`（默认）：允许部分跳转携带。
- **作用：有效防御 CSRF 攻击。**

**`Domain / Path`**：限制 Cookie 的生效范围。





#### LocalStorage（本地存储）

- **场景**：保存用户偏好设置（皮肤、语言）、长期缓存的数据。
- **注意**：如果数据量过大，会阻塞主线程（同步操作），影响页面渲染。

#### SessionStorage（会话存储）

- **场景**：敏感的临时表单数据、单次登录状态。
- **误区**：**只要窗口不关，数据就在。** 刷新页面数据不会丢，但**复制一个新标签页打开**，即便 URL 一样，SessionStorage 也是不共享的（除非是从当前页 `window.open` 出来的）。





## 为什么0.1+0.2不等于0.3

在 JavaScript（以及大多数编程语言）中，`0.1 + 0.2` 的结果是 `0.30000000000000004`。

这并不是因为 JS 算错了，而是由于**计算机底层存储数字的方式**决定的。我们可以从以下三个逻辑层面来拆解这个现象：

------

### 1. 进制转换的“无限循环”问题

人类使用 **十进制**（以 10 为基数），而计算机使用 **二进制**（以 2 为基数）。

- 在十进制中，像 $1/3$ 这样的数字是无法除尽的（$0.3333...$）。
- 在二进制中，**$0.1$ 和 $0.2$ 也是无法除尽的**。

当计算机尝试把 $0.1$ 转换成二进制时，它会变成一个无限循环小数：

$0.1 \rightarrow 0.00011001100110011... (1100 循环)$

------

### 2. IEEE 754 标准的“强行截断”

计算机的内存是有限的，不能存储无限长度的小数。JavaScript 遵循 **IEEE 754 标准**，使用“双精度浮点数”（64位）来存储数字。

由于位数有限（只有 52 位用于存储尾数），计算机必须对无限循环的二进制进行**舍入（Rounding）**。

- 这就好比你想在纸上写下 $1/3$，但纸的宽度只能写 10 位，你只能写下 $0.3333333333$。
- **误差就在这一刻产生了。**

------

### 3. 运算后的累加误差

当你执行 `0.1 + 0.2` 时：

1. 计算机先取出截断后的 `0.1` 二进制。
2. 取出截断后的 `0.2` 二进制。
3. 将这两个带有微小误差的数字相加。
4. 得到的结果再转换回十进制，就变成了 `0.30000000000000004`。

------

### 🛠️ 面试中如何解决这个问题？

如果面试官问：“在前端开发中如何避免这种精度问题？”你可以给出以下方案：

#### 方案 A：转换为整数（推荐）

先放大成整数运算，再缩小回去。

JavaScript

```
(0.1 * 10 + 0.2 * 10) / 10; // 0.3
```

#### 方案 B：使用 `toFixed()`

四舍五入保留指定位数，注意返回的是字符串。

JavaScript

```
parseFloat((0.1 + 0.2).toFixed(12)); // 0.3
```

#### 方案 C：设置“误差范围” (Epsilon)

ES6 提供了 `Number.EPSILON`，它表示 1 与大于 1 的最小浮点数之间的差。如果两个数的差值小于这个数，我们就认为它们相等。

JavaScript

```
function numbersAreEqual(n1, n2) {
  return Math.abs(n1 - n2) < Number.EPSILON;
}
console.log(numbersAreEqual(0.1 + 0.2, 0.3)); // true
```

#### 方案 D：使用专业库

如果是涉及金融计算，使用 `Big.js`、`Decimal.js` 或 `Math.js`。
