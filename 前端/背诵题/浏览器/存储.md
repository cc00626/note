| **特性**       | **Cookie**                             | **LocalStorage**                | **SessionStorage**           |
| -------------- | -------------------------------------- | ------------------------------- | ---------------------------- |
| **生命周期**   | 可设置过期时间，否则默认关闭浏览器失效 | **永久保存**，除非手动删除      | **仅在当前会话（窗口）**有效 |
| **存储大小**   | 非常小（约 **4KB**）                   | 较大（约 **5MB**）              | 较大（约 **5MB**）           |
| **服务端通信** | **自动携带**在 HTTP 请求头中           | 仅在客户端保存                  | 仅在客户端保存               |
| **数据操作**   | 比较麻烦（需自行封装）                 | 原生 API（`setItem`/`getItem`） | 原生 API                     |

**`HttpOnly`**：**最重要**。设置后，JS 无法通过 `document.cookie` 读取。**能有效防止 XSS 攻击盗取 Cookie。**

**`Secure`**：只在 HTTPS 协议下传输。

**`SameSite`**：

- `Strict`：完全禁止第三方 Cookie。
- `Lax`（默认）：允许部分跳转携带。
- **作用：有效防御 CSRF 攻击。**

**`Domain / Path`**：限制 Cookie 的生效范围。



### LocalStorage（本地存储）

- 跨域

- **场景**：保存用户偏好设置（皮肤、语言）、长期缓存的数据。
- **注意**：如果数据量过大，会阻塞主线程（同步操作），影响页面渲染。

### SessionStorage（会话存储）

- **场景**：敏感的临时表单数据、单次登录状态。
- **误区**：**只要窗口不关，数据就在。** 刷新页面数据不会丢，但**复制一个新标签页打开**，即便 URL 一样，SessionStorage 也是不共享的（除非是从当前页 `window.open` 出来的）。



### localStorage一次性插入大小限制的数据怎么进行解决的

1. 终极方案：切换到 IndexedDB (推荐)

如果你的数据量很大（比如几百 MB），`localStorage` 的架构已经不适用了。**IndexedDB** 是浏览器内置的非关系型数据库，几乎没有容量限制（通常可占磁盘可用空间的 50% 以上）。

- **优点**：支持索引、支持二进制存储（Blob/ArrayBuffer）、异步操作不阻塞 UI。
- **工具推荐**：原生 API 比较复杂，建议使用 **localForage** 库。它的 API 和 `localStorage` 几乎一模一样，但底层跑在 IndexedDB 上。

JavaScript

```
import localforage from 'localforage';

// 像使用 localStorage 一样，但它是异步的
await localforage.setItem('largeData', bigObject);
const value = await localforage.getItem('largeData');
```

------

2. 数据压缩方案 (LZ-String)

如果你的数据只是稍微超出了一点（比如 6MB），或者你由于某些原因必须使用 `localStorage`，可以使用文本压缩算法。

- **原理**：利用 **LZ-String** 等库对字符串进行压缩后再存入。
- **效果**：通常可以将 JSON 字符串压缩至原大小的 **20%~50%**。

JavaScript

```
import LZString from 'lz-string';

const compressed = LZString.compress(JSON.stringify(hugeData));
localStorage.setItem('data', compressed);

// 读取时解压
const decompressed = LZString.decompress(localStorage.getItem('data'));
```

------

3. 跨域/多域名“分库”方案

`localStorage` 的限制是基于 **协议+域名+端口** 的。

- **策略**：你可以利用 `iframe` 嵌入其他子域名的页面，通过 `postMessage` 通信，将数据分散存储在不同域名下的 `localStorage` 中。
- **场景**：这种方案比较“黑产”风格，维护成本高，仅在极端特殊环境下使用。

------

4. 数据清理与分片

如果数据是日志或缓存，可以从管理策略上解决：

- **LRU 算法**：实现一个“最近最少使用”淘汰机制。当发现空间不足时，自动删除最旧的数据。
- **数据切片**：将一个巨大的 Key 拆分成 `data_part1`, `data_part2`。虽然总限额没变，但可以避免单个 Value 过大导致的写入失败。
