## 输入url到页面出现的过程

1. 解析url，如果url符合规范，会尽量拼接成完整的url，比如在前端加上http：// ，如果url不符合规范搜索引擎会将内容作为关键字，然后使用搜索引擎搜索结果的url。

2. 检查缓存，发送请求前会先检查缓存中是否有强缓存，检查expires和cache-control中的时间，查看资源是否过期，强缓存无效就检查协商缓存，通过if-none-match 和if-modeify-since 带上最后的修改时间，服务端进行比较，决定是否使用。

3. DNS域名解析，根据域名找到对应的ip地址，先去浏览器的缓存中寻找，然后去操作系统中，host文件中寻找，然后进行迭代查询，根域名 --- 顶级域名服务器--- 权威域名服务器。最后找到对应的ip地址。这个过程发生在应用层。

4. 进行tcp连接和三次握手

5. 发送http或者https请求，浏览器会构建请求行，请求头，请求体，如果是https构建之前要进行非对称加密和对称加密

6. 接受响应，如果是301永久重定向，302会临时重定向，检查connectiontype ： keep-alive是否是长连接，如果不是就进行四次挥手，服务器会返回一个html文件。浏览器开始解析html文件。

7. 解析html，从上到下解析html，将html解析成dom树，如果遇到css代码，则构建css树，遇到script会阻止html的解析，除非带上defer或者async，

8. 浏览器在解析 HTML 构建 DOM Tree，解析 CSS 构建 CSSOM Tree 之后，会进行 Style Calculation，为每一个 DOM 节点计算最终的 Computed Style。然后浏览器会将 DOM Tree 和 CSSOM Tree 合并，构建 Render Tree。Render Tree 只包含需要渲染的节点，例如 display:none 的元素不会进入 Render Tree，但 visibility:hidden 和 opacity:0 的元素仍然会进入，因为它们仍然参与布局。

   Render Tree 中的每个节点包含元素的样式信息，例如盒模型、位置属性等，之后浏览器会进入 Layout 阶段计算元素的位置和大小。

9. 布局，从根对象中递归处理，计算每个dom的大小和位置，这个过程也成为回流，构成布局树。

10. 分层

- 有 `position: fixed / sticky`
- 有 `will-change`、`transform`、`opacity` 动画
- 有 `z-index` 或者 3D 特性

使每个图层都可以单独进行处理

10. 绘制，将分层过后的元素生成一些指令，绘制在屏幕上，主要进行像素的填充，比如颜色

11. 分块，分成一个的小块。

12. 栅格化，将分成的每一个小块构建成像素，优先处理视口内的内容,光栅化的结果，就是一块一块的位图

13. 所有的图层经过栅格化后，**由 GPU 合成到一张最终图像中**，
    这个阶段执行各种 **transform、opacity 等视觉变化**，
    最后交由显示设备呈现。  

14. 四次握手。





## 为什么 transform 的效率高？

因为 transform 既不会影响布局也不会影响绘制指令，它影响的只是渲染流程的最后一个「draw」阶段

由于 draw 阶段在合成线程中，所以 transform 的变化几乎不会影响渲染主线程。反之，渲染主线程无论如何忙碌，也不会影响 transform 的变化。

## 什么是重绘和重排？如何减少

重排（Reflow）是浏览器重新计算元素的几何信息，比如宽高、位置等。当元素的布局属性发生变化时就会触发重排，比如修改 width、height、margin、position 等。

重绘（Repaint）是元素样式发生变化，但不影响布局，例如 color、background 等，只需要重新绘制。



重排一定会触发重绘，但重绘不一定触发重排。



1. 减少dom操作，如果需要插入dom，可以放入一个片段中，一次性插入
2. 使用 transform 代替 top/left
3. 避免频繁读取布局信息，如 offsetWidth，使用ResizeObserver，IntersectionObserver代替
4. 使用will-change
5. 使用 requestAnimationFrame 优化动画

**will-change优化的原理**

will-change 是一个 CSS 属性，用来告诉浏览器某个元素的某些属性即将发生变化，例如 transform 或 opacity。浏览器可以提前为该元素创建独立的合成图层，从而避免动画开始时再进行图层创建，提高动画的渲染性能。

使用 will-change 可以减少动画过程中产生的重排和重绘，让动画更多地在 GPU 合成阶段完成。

不过 will-change 也不能滥用，因为每个图层都会占用 GPU 内存，如果大量使用会导致浏览器性能下降。通常建议只在动画元素上使用，并在动画结束后移除。



**requestAnimationFrame优化的原理**

requestAnimationFrame 是浏览器提供的动画 API，它会在浏览器下一次重绘之前执行回调函数。

浏览器一般以约 16.7ms（60FPS）为一帧进行渲染，而 requestAnimationFrame 会与这个刷新节奏保持同步。

使用 requestAnimationFrame 时，DOM 的多次修改可以在同一帧内被浏览器合并处理，只触发一次 Layout 和 Paint，从而减少不必要的重排和重绘。

相比 setTimeout 或 setInterval，requestAnimationFrame 还能在页面不可见时自动暂停执行，从而节省 CPU 资源，因此更适合用于实现动画。
