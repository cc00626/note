## react和vue的区别

### 1. 事件绑定不同

- vue的事件绑定，是绑定到使用了v-on的具体元素上，并且事件回调函数的event参数是原生的 

- React拥有自己的合成事件机制，事件绑定是默认使用了事件委托的，React17版本之前会绑定到 document上面，React17版本以后事件绑定到了root节点上，并且事件回调函数的event参数是 React重新封装后的

### 2. diff的更新策略不同

### vue

#### 1. 入口阶段：`patch` 函数

当响应式数据发生变化时，Vue 会触发 `patch` 过程，这是 Diff 的起点。

- **判定相同 (`sameVnode`)：** 检查新旧节点的 `key`、`tag`、`isComment` 等属性是否一致。
- **分流处理：**
  - **不同：** 说明节点类型已变，直接销毁旧 DOM，创建并挂载新 DOM。
  - **相同：** 触发 `patchVnode` 逻辑，进入“精细化对比”阶段。

------

#### 2. 精细化对比：`patchVnode`

在这个阶段，Vue 开始处理节点的具体细节：

| **场景**           | **处理方式**                                                |
| ------------------ | ----------------------------------------------------------- |
| **静态节点**       | 如果节点是静态的（无动态绑定），直接跳过，节省开销。        |
| **文本节点**       | 若新旧节点仅为文本且内容不同，直接执行 `textContent` 更新。 |
| **子节点对比**     | 这是最核心的情况，分为以下三类：                            |
| * 只有新节点有子级 | **挂载**：直接新建并插入子节点。                            |
| * 只有旧节点有子级 | **卸载**：直接删除旧的子节点。                              |
| * 两者都有子级     | **更新**：进入最复杂的 `updateChildren` 流程。              |

------

#### 3. 子节点更新：`updateChildren` (双端比较)

Vue 2 和 Vue 3 的核心对比逻辑都基于**双端比较**算法。通过维护四个指针，尽可能地寻找可复用的节点：

- **四个指针：** `oldStart` (旧头)、`oldEnd` (旧尾) 与 `newStart` (新头)、`newEnd` (新尾)。

**比较优先级（命中即跳出）：**

1. **头对头：** 相同则复用，指针向中间靠拢。
2. **尾对尾：** 相同则复用，指针向中间靠拢。
3. **头对尾：** 旧头移动到旧尾之后，旧头指针右移，新尾指针左移。
4. **尾对头：** 旧尾移动到旧头之前，旧尾指针左移，新头指针右移。
5. **查找 Key：** 若上述四种均未命中，则根据新节点的 `key` 在旧节点池中查找：
   - **找到：** 复用节点并移动位置。
   - **未找到：** 创建新节点。

------

#### 4. Vue 3 的进阶优化：最长递增子序列 (LIS)

虽然 Vue 3 保留了双端预处理，但在处理“乱序”节点的复杂场景时，引入了更高效的算法。

- **核心逻辑：**
  1. **静态提升：** 在编译阶段标记静态内容，Diff 时直接跳过。
  2. **LIS 算法：** 在乱序移动中，通过“最长递增子序列”找到一组**相对位置不需要变动**的节点集合。
- **结果：** 只有不在子序列中的节点才执行 `move` 操作。这极大减少了 DOM 节点的移动次数，提升了复杂列表更新的性能。



### react

React 的子节点 Diff（`reconcileChildrenArray`）主要分为两个阶段：

#### 第一轮遍历：找“位置没变”的节点

React 会从左到右，同时遍历新旧两个子节点数组。

- **对比方式**：比较 `oldFiber[i]` 和 `newVNode[i]`。
- **命中复用**：如果 `key` 和 `type` 都相同，则复用旧节点，继续对比下一个。
- **跳出条件**：
  1. **Key 不同**：一旦发现 `key` 不匹配，立即停止第一轮遍历。
  2. **任一数组遍历完**：如果旧节点先遍历完，新节点剩下的直接插入；如果新节点先遍历完，旧节点剩下的直接删除。

------

#### 第二轮遍历：找“位置变了”的节点

如果第一轮遍历因为 `key` 不匹配而中断，说明剩下的节点里存在**位移、新增或删除**。

#### 1. 建立 Map 映射表

React 会把剩余的所有**旧节点**放入一个 `Map` 中：

- **Key**：节点的 `key`（如果没有 key 则使用索引 `index`）。
- **Value**：旧的 Fiber 节点。

#### 2. 遍历剩余的新节点

React 拿着剩下每一个新节点，去 `Map` 里找有没有匹配的 `key`。

#### 3. 核心机制：`lastPlacedIndex` (判断是否移动)

这是 React Diff 最精妙的地方。它维护了一个变量 `lastPlacedIndex`，记录**当前已复用节点在旧集合中的最大索引**。

- **逻辑**：
  - 设当前新节点在旧集合中的原始索引为 `oldIndex`。
  - 如果 `oldIndex < lastPlacedIndex`：说明该节点**在旧集合中原本在左边**，现在要移到当前位置的右边。**标记为“移动”**。
  - 如果 `oldIndex >= lastPlacedIndex`：说明该节点不需要移动。更新 `lastPlacedIndex = oldIndex`。

## Vue2和Vue3 diff算法的差异

Vue2 的 diff 算法主要使用双端比较策略，通过 oldStart、oldEnd、newStart、newEnd 四个指针进行节点比较，在节点乱序较多的情况下需要遍历查找 key，最坏时间复杂度可能达到 O(n²)。

Vue3 在此基础上进行了多项优化。首先在 diff 前后阶段通过头尾指针快速跳过相同节点，其次在处理中间乱序节点时使用 key-index map 提高查找效率，并通过最长递增子序列算法减少 DOM 移动次数。同时 Vue3 在编译阶段引入 PatchFlag 和 Block Tree 机制，只追踪动态节点，从而大幅减少不必要的 diff 操作，因此整体性能相比 Vue2 有明显提升。



## react16，17，18，19的区别

**react16**

- 引入Fiber架构，虽然将渲染任务拆分成了小的单元（Work In Progress），但在默认情况下，它的执行逻辑依然是**“同步执行”**的，引入react Hooks(16.8)
- 引入错误边界。
- 支持返回多个根节点，减少无意义的 HTML 层级



**react17**

- 事件委托从docuemtn转到放在应用根节点，



**react18**

- 并发模式

  在 React 18 之前，渲染是不可中断的（Stack Reconciler）。一旦开始渲染大组件，主线程就会被阻塞，用户点击或输入都没反应

- 自动批处理

  以前：只有 React 事件处理函数会批处理。

  18：`setTimeout`、`Promise`、原生事件等产生的多个 `setState` 都会**自动合并**为一次渲染。

- Transitions 

  引入 `useTransition` 和 `useDeferredValue`。

  区分**紧急更新**（如打字、点击）和**非紧急更新**（如搜索列表展示），确保高优先级交互不卡顿。

- **Suspense 增强**：支持服务端渲染（SSR）下的流式渲染（Streaming SSR），开启html的分块传输，renderToPipeableStream

  A. HTML 流式传输 (Streaming HTML)

  服务器不再等所有数据。你可以把加载慢的组件包裹在 `<Suspense>` 中。

  - **第一步**：服务器先发送页面的框架（如 Header、导航栏）和 Suspense 的占位符（Loading 状态）。
  - **第二步**：浏览器立即渲染出这些初始 HTML。
  - **第三步**：当服务器后台拿到了慢组件的数据，它会生成该组件的 HTML，通过**同一个 HTTP 连接**补发给浏览器，并自动替换掉之前的 Loading 占位符。

  B. 选择性注水 (Selective Hydration)

  这是最聪明的地方。React 18 允许在 JS 还没全下载完时，**先给已经渲染好的部分注水**。

  - 如果 Header 的 JS 先到了，Header 就可以先点。
  - **交互优先**：如果用户点击了还在 Loading 的“评论区”，React 会意识到用户很急，它会优先下载评论区的 JS 并优先给它注水，这种“按需分配”极大提升了用户体感。

- useId生成唯一 ID



**react19**

**Actions (动作)**：

- 原生支持处理异步函数。通过 `useActionState` 自动管理 `pending`、`error` 和响应状态，不再需要手动维护一大堆 `loading` 状态。

  **`useActionState`**：这是核心 Hook。你传给它一个异步函数，它还给你一个 `isPending`（自动变 true/false）和最新的结果。

  **不再需要手动 `setLoading`**：只要你的函数是 `async` 的，React 会追踪它的生命周期。当函数在 `await` 时，`isPending` 自动为 `true`；执行完自动变 `false`。

**useOptimistic (乐观更新)**：

- 专门用于处理“先修改 UI，后台失败再回滚”的场景，极大简化了社交点赞等功能的代码。

**新的 API：`use`**：

**意思**：它是一个可以在组件渲染过程中“读取”资源的值的 API。

**打破规则**：以往的 Hook（如 `useEffect`）绝对不能写在 `if` 或 `for` 里。但 `use` **可以**写在条件判断里！

**用法**：

- **读 Promise**：`const data = use(fetchPromise)`。如果 Promise 没完，React 自动触发最近的 `<Suspense>`。
- **读 Context**：`const theme = use(ThemeContext)`。如果你只需要在某种特定条件下用 Context，现在可以把它塞进 `if` 语句里，非常灵活。

**Ref 简化**：不再需要 `forwardRef`！现在可以直接把 `ref` 作为 props 传递给子组件。

**React Compiler (编译器)**：

- *重大变革*：自动管理 Memoization。以后你可能不再需要写 `useMemo` 和 `useCallback` 了，React 编译器会在构建阶段自动帮你处理。
