### 为什么要推出组合式APi

**关注点被拆散**

在 Vue 2 的 Options API 中，我们是按照**属性**来组织代码的：`data` 放在一处，`methods` 放在一处，`computed` 又在一处。

当一个组件变得很大（比如 1000 行）时，如果你要修改一个“搜索功能”，你需要反复在代码里“上下蹦迪”：去 `data` 看变量，跳到 `methods` 改函数，再跳到 `watch` 看监听。

 它允许你将**同一个功能**的相关代码（数据、方法、生命周期）紧凑地写在一起。就像把乱七八糟的零件组装成了功能模块，阅读和维护效率翻倍。



**实现更好的代码复用**

在 Vue 2 中，跨组件复用逻辑主要靠 **Mixins (混入)**，但它简直是开发者的噩梦：

- **命名冲突：** 两个 Mixin 里都有 `name` 属性，谁覆盖谁？
- **来源不明：** 看到一个 `this.getData()`，你翻遍当前文件找不到，最后发现它藏在第三个 Mixin 里。
- **组合式 API 的解法：** 通过普通的 JavaScript 函数（Hook 模式）导出和导入。数据来源清晰，且可以利用解构赋值轻松重命名，完美解决了 Mixins 的所有弊端。



**树摇优化（Tree-shaking）**

- **Vue 2：** 很多功能是直接挂在 Vue 实例上的。即便你没用到 `nextTick` 或某些内置组件，它们也会被打包进去。
- **Vue 3 (组合式)：** 所有的 API 都是通过 `import { ref, onMounted } from 'vue'` 引入的。如果你没用到某个 API，打包工具（如 Vite/Webpack）就会自动把它删掉，从而减小最终产物的体积。
