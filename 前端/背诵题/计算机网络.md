## 网络模型

### 1. 概念分层：为什么要分层？

网络通信极其复杂（涉及电信号、路由选择、数据纠错、应用协议）。分层设计的核心目的就是**“解耦”**：每一层只负责自己的任务，并为上层提供服务。只要接口不变，底层技术的更替（如从电缆换成光纤）不会影响上层应用（如网页浏览）。

### 2. OSI 七层模型 vs TCP/IP 四层模型

#### 第一层：物理层 (Physical Layer)

- **功能：** 传输原始的比特流（0 和 1）。它定义了电压、接口规格、传输速率。
- **设备：** 中继器、集线器、光纤。
- **快递比喻：** 物理层就是**运输工具**（卡车、飞机）和**道路**。

#### 第二层：数据链路层 (Data Link Layer)

- **功能：** 将比特流组合成**帧 (Frame)**。负责物理寻址（**MAC 地址**）和差错检测。
- **核心协议：** 以太网 (Ethernet)、ARP (地址解析协议)。
- **快递比喻：** 就像是在省际分拨中心，通过**车牌号**识别下一站去哪。

#### 第三层：网络层 (Network Layer)

- **功能：** 负责路由选择和逻辑寻址（**IP 地址**）。它决定了数据包从 A 到 B 走哪条路最快。
- **核心协议：** **IP**、ICMP (Ping 命令用的协议)、OSPF。
- **快递比喻：** 快递单上的**收件人地址**。分拣中心根据地址规划运输路线。

#### 第四层：传输层 (Transport Layer)

- **功能：** 负责**端到端**的通信。它决定了数据是可靠传输（TCP）还是快速传输（UDP）。
- **核心协议：** **TCP**（可靠、三次握手）、**UDP**（不可靠、快）。
- **快递比喻：** 快递的**服务类型**。TCP 是“挂号信”（丢了必赔，必须签字确认）；UDP 是“平信”（寄出去就不管了）。

#### 第五/六/七层：应用层 (Application Layer)

*在 TCP/IP 模型中，这三层被合并为一层。*

- **会话层：** 建立、管理和终止表示层实体之间的会话。
- **表示层：** 数据格式化、加密解密（如 SSL/TLS）、压缩。
- **应用层：** 直接面向用户。协议有 **HTTP、DNS、FTP、SMTP** 等。
- **快递比喻：** 快递盒里的**具体物品**。是你写的信，还是你寄的苹果？

------

### 3. 关键过程：封装与解封装

这是理解网络模型的精髓。数据在发送端是**“加头”**的过程，在接收端是**“去头”**的过程。

1. **应用层**产生原始数据。
2. **传输层**加上 TCP 头（包含端口号），变成 **Segment（段）**。
3. **网络层**加上 IP 头（包含 IP 地址），变成 **Packet（包）**。
4. **数据链路层**加上 MAC 头，变成 **Frame（帧）**。
5. **物理层**将其变成电信号发出。

## HTTP/1.0、HTTP/1.1 与 HTTP/2.0 区别

### HTTP/1.0 vs HTTP/1.1

#### 1. 连接管理

- **HTTP/1.0**

- 默认采用非持久连接。
- 每发送一个请求都要建立新的 TCP 连接，响应完成后即关闭连接。

- **HTTP/1.1**

- 默认启用持久连接（Keep-Alive）。
- 通过 `Connection: keep-alive` 控制连接保持，允许在同一 TCP 连接上发送多个请求。
- 连接中请求必须按顺序处理（串行执行），不支持真正的并发请求。
- 浏览器为了提升性能，限制同一域名下最大并发 TCP 连接数为 6~8 个（现代浏览器通常为 6 个）。

------

#### 2. 资源请求

- **HTTP/1.0**

- 不支持断点续传，客户端只能请求整个资源。
- 下载过程中断时必须重新从头开始下载，造成带宽浪费。

- **HTTP/1.1**

- 引入 `Range` 请求头，允许客户端请求资源的部分字节范围。
- 服务器收到带有 Range 请求时返回状态码 `206 Partial Content`，仅返回指定范围的数据。
- 支持断点续传，大大节省带宽和时间，提高用户体验。

------

#### 3. 缓存机制

- **HTTP/1.0**

- 主要使用请求头 `If-Modified-Since` 和响应头 `Expires` 控制缓存。
- `If-Modified-Since`：客户端带上资源的最后修改时间，服务器对比决定是否返回资源。**（协商缓存）**
- `Last-Modified`：服务器告诉客户端资源最后修改时间。
- `Expires`：服务器告诉客户端缓存的过期时间，过期前优先使用缓存。**（强缓存）**

- **HTTP/1.1**

- 使用 **ETag** 和更多缓存控制头部。
- `Cache-Control ` : **（强缓存）**
- `ETag`：资源唯一标识（指纹），资源内容变更时 ETag 改变。
- `If-None-Match`：客户端带上缓存资源的 ETag，匹配则返回 `304 Not Modified`。**（协商缓存）**

------

### 4. Host 请求头

- **背景**

- HTTP/1.0 假设每台服务器绑定唯一 IP，客户端请求时不携带主机名。
- 随着虚拟主机技术发展，一台服务器可托管多个网站。

- **问题**

- HTTP/1.0 请求没有主机名字段，服务器无法区分请求目标虚拟主机。

- **解决方案（HTTP/1.1）**

- 引入 `Host` 请求头，客户端必须携带目标服务器域名。
- 服务器根据 `Host` 字段路由请求，实现多个网站共享同一 IP。

- **示例请求**

```plain
http


复制编辑
GET /index.html HTTP/1.1
Host: www.example.com
```

------

### 5. 管道化（Pipelining）

- **HTTP/1.1** 的性能优化技术。
- 客户端可以一次性发出多个请求，无需等待前一个响应。
- 服务器按顺序处理请求并返回响应，提高连接效率。

------

## HTTP/1.1 vs HTTP/2

### 1. 传输格式：文本 vs 二进制 (The Core Change)

这是 H2 性能提升的基石。

- **HTTP/1.1 (文本协议)：** 报文由 Header（文本）和 Body（文本或二进制）组成，中间用换行符分割。
  - **缺点：** 计算机解析文本效率低，且文本存在歧义（如换行符处理），健壮性差。
- **HTTP/2 (二进制分帧)：** 将所有传输信息分割为更小的**帧（Frame）**，并采用二进制格式编码。
  - **帧与流：** 头部信息被封装在 `HEADERS frame` 中，数据封装在 `DATA frame` 中。这些帧组合成一个逻辑上的“流（Stream）”。
  - **优势：** 二进制解析更快、更紧凑，错误率更低。

------

### 2. 多路复用：彻底解决队头阻塞 (Multiplexing)

这是 H2 最著名的特性。

- **HTTP/1.1 (队头阻塞 HOLB)：**
  - 虽然支持长连接，但在同一个 TCP 连接里，请求必须是**串行**的。
  - 如果第一个请求（如大图）卡住了，后面的请求（如关键 CSS）必须排队等死。
  - **老土的优化手段：** 以前我们会用“域名收敛”（开 6 个子域名并行下载）或“精灵图（Sprite）”来规避这个问题。
- **HTTP/2 (多路复用)：**
  - 在同一个 TCP 连接上，客户端可以**同时**发出无数个请求，服务器也可以**并行**返回响应。
  - 不同请求的“帧”混杂在一起传输，通过帧头部的 **Stream ID** 来标识属于哪个请求。
  - **结果：** 真正实现了单连接下的全双工通信，彻底消除了应用层的队头阻塞。

------

### 3. 头部压缩：告别冗余 (HPACK)

HTTP 是无状态的，这意味着每次请求都要带上相同的 Header。

- **HTTP/1.1 (重复传输)：** 每次请求都要携带 `User-Agent`、`Cookie`、`Accept` 等几百甚至上千字节的文本。在一个有 100 个请求的页面里，光 Header 就要浪费几十 KB 的带宽。
- **HTTP/2 (HPACK 算法)：**
  - **静态表：** 预定义了 61 个高频字段（如 `GET`、`200` 等），只用一个数字索引表示。
  - **动态表：** 通信双方共同维护一张表。如果第一次发了完整的 `User-Agent`，第二次只需发个索引号，或者只发送“增量”部分。
  - **Huffman 编码：** 对字符串进行哈希压缩。
  - **效果：** Header 的体积通常能减少 **80% - 90%**。

------

### 4. 服务器推送 (Server Push)

- **HTTP/1.1 (被动)：** 浏览器解析完 HTML，发现里面有 `style.css`，再去发请求问服务器要。
- **HTTP/2 (主动)：** 服务器“预判”了你的预判。当你请求 `index.html` 时，服务器知道你肯定需要 CSS 和 JS，于是直接把这些资源顺着连接推送到浏览器的缓存里。
  - **优势：** 减少了等待浏览器解析 HTML 并重新发起请求的 RTT（往返时间）。

------

### 5. 请求优先级 (Stream Priority)

由于 H2 是多路复用的，如果所有资源一起抢占带宽，可能会导致关键的 CSS 还没下完，无关紧要的图片先下好了。

- **HTTP/2：** 允许客户端给每个“流”设置权重和依赖关系。浏览器会告诉服务器：“优先给我发这个 JS 脚本，图片可以慢慢传”。服务器会根据优先级分配带宽资源。



## HTTP2.0存在的问题

### 1. TCP 层的队头阻塞 (The Biggest Flaw)

这是 H2 最核心的问题。H2 的多路复用是在一个 TCP 连接上跑多个 Stream。

- **机制：** TCP 是一个“可靠传输”协议，它必须保证数据包按顺序到达。
- **问题：** 如果其中一个 Stream 的某个数据包在网络传输中**丢失**了，TCP 就会卡住所有后面的包，进行重传。
- **结果：** 即使剩下的 Stream（比如图片 B、脚本 C）的数据包已经平安到达了浏览器的缓冲区，但因为前面的图片 A 丢了一个包，TCP 不允许应用层读取后面的数据。
- **讽刺点：** 在丢包严重的网络环境下，**HTTP/2 的表现甚至不如 HTTP/1.1**（因为 1.1 通常开 6 个连接，一个丢包只影响其中一个，另外 5 个还能跑）。

------

### 2. 握手延迟 (Handshake Latency)

H2 依然建立在 **TCP + TLS** 之上。

- **繁琐过程：**
  1. TCP 三次握手（1.5 RTT）
  2. TLS 握手（通常 1-2 RTT）
- **痛点：** 在发送第一个字节的业务数据之前，客户端和服务器已经来回“确认眼神”好几次了。在移动网络（高延迟）环境下，这种延迟非常明显。

------

### 3. 对 CPU 消耗更高

H2 的高性能是用“算力”换来的。

- **原因：** 二进制分帧、HPACK 头部压缩算法、多路复用后的流管理，这些都需要客户端和服务器花费额外的 CPU 来进行编解码和状态维护。
- **现状：** 虽然现代设备性能很强，但在超高并发的服务器端，H2 的维护成本确实比简单的文本协议 1.1 要高。

------

### 4. 强制加密 (De Facto Standard)

虽然 HTTP/2 标准本身并不强制要求加密（可以是 `h2c`），但主流浏览器（Chrome, Firefox, Safari）一致决定：**只支持基于 TLS 的 HTTP/2**。

- 这意味着你必须部署证书，且无法逃避 TLS 带来的加解密开销。

## 强缓存和协商缓存的区别

### 强缓存

**Cache-Control (HTTP/1.1，优先级最高)：** * `max-age=3600`：表示资源在 3600 秒内有效。

- `public/private`：决定中间代理服务器（如 CDN）是否可以缓存。
- `no-cache`：**大坑点！** 它的意思不是不缓存，而是**必须走协商缓存**（先问服务器）。
- `no-store`：这才是真正的完全不缓存。

**Expires (HTTP/1.0，优先级较低)：**

- 使用绝对时间（如 `Wed, 21 Oct 2025 07:28:00 GMT`）。
- **缺点：** 依赖客户端本地时钟，如果用户电脑时间调错了，缓存就失效了。

### 协商缓存

#### 时间戳（最后修改时间）

- **Last-Modified (响应头)**：服务器告诉浏览器：“我这文件最后修改是 10:00”。
- **If-Modified-Since (请求头)**：下次请求时，浏览器问：“10:00 以后你改过吗？”
  - **没改：** 服务器返回 **304 Not Modified**，不发数据体，省流量。
  - **改了：** 服务器返回 **200 OK** 和最新的文件。

#### 指纹（内容哈希，优先级更高）

- **ETag (响应头)**：服务器给文件算一个唯一的“指纹”（类似 Hash）。
- **If-None-Match (请求头)**：浏览器问：“你的指纹还是这个 ABC 吗？”
  - **对比一致：** 返回 **304**。
  - **不一致：** 返回 **200**。



在实际的项目部署（如 Nginx 配置）中，我们通常采用以下策略：

1. **HTML：使用协商缓存**
   - 设置 `Cache-Control: no-cache`。
   - 保证每次打开网页都能向服务器确认 HTML 是否有更新。
2. **静态资源 (JS/CSS/图片)：使用强缓存 + Hash 名**
   - 设置 `Cache-Control: max-age=31536000` (一年)。
   - **关键点**：文件名带上内容 Hash（如 `main.a1b2c3.js`）。如果代码改了，文件名就变了，浏览器会当成新资源去下载，解决了“强缓存无法更新”的问题。



## TCP和UDP

### TCP概念和特点

TCP的全称是传输控制协议是一种面向连接的、可靠的、基于字节流的传输层通信协议。TCP 是面向连接的、可靠的流协议（流就是指不间断的数据结构）。

**1）面向连接**

面向连接，是指发送数据之前必须在两端建立连接。建立连接的方法是“三次握手”，这样能建立可靠的连接。建立连接，是为数据的可靠传输打下了基础。

**2）仅支持单播传输**

每条TCP传输连接只能有两个端点，只能进行点对点的数据传输，不支持多播和广播传输方式。

**3）面向字节流**

TCP不像UDP一样那样一个个报文独立地传输，而是在不保留报文边界的情况下以字节流方式进行传输。

**4）可靠传输**

对于可靠传输，判断丢包、误码靠的是TCP的段编号以及确认号。TCP为了保证报文传输的可靠，就给每个包一个序号，同时序号也保证了传送到接收端实体的包的按序接收。然后接收端实体对已成功收到的字节发回一个相应的确认(ACK)；如果发送端实体在合理的往返时延(RTT)内未收到确认，那么对应的数据（假设丢失了）将会被重传。

**5）提供拥塞控制**

当网络出现拥塞的时候，TCP能够减小向网络注入数据的速率和数量，缓解拥塞。

**6）提供全双工通信**

TCP允许通信双方的应用程序在任何时候都能发送数据，因为TCP连接的两端都设有缓存，用来临时存放双向通信的数据。当然，TCP可以立即发送一个数据段，也可以缓存一段时间以便一次发送更多的数据段（最大的数据段大小取决于MSS）



### UDP的概念和特点

UDP的全称是**用户数据报协议**，在网络中它与TCP协议一样用于处理数据包，是一种无连接的协议。在OSI模型中，在传输层，处于IP协议的上一层。UDP有不提供数据包分组、组装和不能对数据包进行排序的缺点，也就是说，当报文发送之后，是无法得知其是否安全完整到达的。

**1）面向无连接**

首先 UDP 是不需要和 TCP一样在发送数据前进行三次握手建立连接的，想发数据就可以开始发送了。并且也只是数据报文的搬运工，不会对数据报文进行任何拆分和拼接操作。

具体来说就是：

- 在发送端，应用层将数据传递给传输层的 UDP 协议，UDP 只会给数据增加一个 UDP 头标识下是 UDP 协议，然后就传递给网络层了
- 在接收端，网络层将数据传递给传输层，UDP 只去除 IP 报文头就传递给应用层，不会任何拼接操作

**2）有单播，多播，广播的功能**

UDP 不止支持一对一的传输方式，同样支持一对多，多对多，多对一的方式，也就是说 UDP 提供了单播，多播，广播的功能。

**3）面向报文**

发送方的UDP对应用程序交下来的报文，在添加首部后就向下交付IP层。UDP对应用层交下来的报文，既不合并，也不拆分，而是保留这些报文的边界。因此，应用程序必须选择合适大小的报文

**4）不可靠性**

首先不可靠性体现在无连接上，通信都不需要建立连接，想发就发，这样的情况肯定不可靠。

并且收到什么数据就传递什么数据，并且也不会备份数据，发送数据也不会关心对方是否已经正确接收到数据了。



再者网络环境时好时坏，但是 UDP 因为没有拥塞控制，一直会以恒定的速度发送数据。即使网络条件不好，也不会对发送速率进行调整。这样实现的弊端就是在网络条件不好的情况下可能会导致丢包，但是优点也很明显，在某些实时性要求高的场景（比如电话会议）就需要使用 UDP 而不是 TCP。

**5）头部开销小，传输数据报文时是很高效的。**

![img](https://cdn.nlark.com/yuque/0/2020/png/1500604/1604023358917-72134998-53bd-4d9c-b71f-7a762dff31a9.png)

UDP 头部包含了以下几个数据：

- 两个十六位的端口号，分别为源端口（可选字段）和目标端口
- 整个数据报文的长度
- 整个数据报文的检验和（IPv4 可选字段），该字段用于发现头部信息和数据中的错误



因此 UDP 的头部开销小，只有8字节，相比 TCP 的至少20字节要少得多，在传输数据报文时是很高效的

## TCP三次握手

### 第一步：客户端发送 SYN (同步)

- **动作**：客户端（Client）发送一个 `SYN=1` 的报文，并随机初始化一个序号 `seq=x`。
- **状态**：客户端进入 `SYN_SENT` 状态。
- **含义**：Client 告诉 Server：“我想和你建立连接，这是我的初始序列号。”

### 第二步：服务端响应 SYN + ACK (同步+确认)

- **动作**：服务端（Server）收到后，回复 `SYN=1`, `ACK=1`。它确认了 Client 的序列号（`ack=x+1`），并给出自己的初始序号 `seq=y`。
- **状态**：服务端进入 `SYN_RCVD` 状态。
- **含义**：Server 告诉 Client：“收到你的请求了，我同意连接。这是我的序列号，你也确认一下。”

### 第三步：客户端确认 ACK (确认)

- **动作**：客户端收到后，回复 `ACK=1`。确认 Server 的序列号（`ack=y+1`），自己的序号变为 `seq=x+1`。
- **状态**：客户端进入 `ESTABLISHED`。服务端收到后也进入 `ESTABLISHED`。
- **含义**：Client 告诉 Server：“我也收到你的确认了。连接正式成立！”



### 为什么是三次

**场景**：如果 Client 发出的第一个 SYN 报文在网络中“迷路”了。Client 超时后重发了第二个 SYN，成功连接并关闭。

**后果**：过了一会儿，那个“迷路”的第一个 SYN 突然到达了 Server。

- 如果只有**两次握手**：Server 只要收到 SYN 就认为连接建立了。此时 Server 会一直等待 Client 发送数据，白白浪费 CPU 和内存资源。
- 如果有**三次握手**：Server 发出 ACK 后，Client 不会理会这个过时的请求，连接就不会建立。



## TCP四次挥手

### 第一步：Client 发送 FIN (我要分手)

- **动作**：客户端发送 `FIN=1`，序号 `seq=u`。
- **状态**：客户端进入 `FIN_WAIT_1`。
- **含义**：Client 说：“我的数据发完了，我想关掉连接。”

### 第二步：Server 发送 ACK (收到，等我一下)

- **动作**：服务端回复 `ACK=1`，确认号 `ack=u+1`。
- **状态**：服务端进入 `CLOSE_WAIT`；客户端收到后进入 `FIN_WAIT_2`。
- **含义**：Server 说：“我知道你想分手了，但我这儿还有点活没干完，你等我把剩下的数据发完。”

### 第三步：Server 发送 FIN (我也好了，分吧)

- **动作**：服务端干完活，发送 `FIN=1, ACK=1`，序号 `seq=v`。
- **状态**：服务端进入 `LAST_ACK`。
- **含义**：Server 说：“我也发完了，咱们正式断开吧。”

### 第四步：Client 发送 ACK (好的，再见)

- **动作**：客户端回复 `ACK=1`，确认号 `ack=v+1`。
- **状态**：客户端进入 `TIME_WAIT`，等待 **2MSL** 后关闭；服务端收到后直接 `CLOSED`。
- **含义**：Client 说：“收到，保重！”

### 为什么是“四次”而不是“三次”？

**这是最高频的追问。**

- **原因**：握手时，Server 的 `SYN` 和 `ACK` 是合并在一起发送的。
- **挥手时**：当 Server 收到 Client 的 `FIN` 时，它可能还有**残留的数据**没发完。它只能先回一个 `ACK` 表示“我知道了”，等数据发完了才能再发 `FIN`。
- **结论**：中间的 `ACK` 和 `FIN` **不能合并**，所以比握手多了一次。

### 为什么 Client 要等 2MSL (最大报文生存时间) 才关？

1. **防止最后一个 ACK 丢失**：如果 Client 最后发的 `ACK` 丢了，Server 会重发 `FIN`。Client 还没关，就能重发 `ACK`。如果 Client 直接关了，Server 就会收到 `RST` 报错，分手就不体面了。
2. **清理“幽灵”报文**：让本次连接产生的旧报文都在网络中消失。防止下次开个一模一样的连接（相同 IP 和端口）时，旧报文跳出来干扰新连接。

> **风险点**：如果服务器上出现大量 `TIME_WAIT`，会占用大量端口资源。通常是因为高并发短连接没处理好。



## cookie的属性和含义

#### 1. 基础属性（控制数据内容）

- **Name & Value**：
  - **含义**：Cookie 的名称和值。
  - **注意**：Value 必须是经过 URL 编码的字符串（因为某些字符在 Cookie 协议中是保留字）。

------

#### 2. 生存周期（控制有效时间）

- **Expires**：
  - **含义**：**绝对过期时间**。由服务器返回一个 GMT 格式的时间点。
  - **特点**：如果达到这个时间点，Cookie 就会被删除。
- **Max-Age**：
  - **含义**：**相对过期时间**（秒）。
  - **优先级**：如果同时存在 `Expires` 和 `Max-Age`，**`Max-Age` 优先级更高**。
  - **分类**：
    - 若不设置这两个属性，该 Cookie 为 **Session Cookie（会话 Cookie）**，浏览器关闭即消失。
    - 若设置了时间，则为 **Persistent Cookie（持久 Cookie）**，存储在磁盘中。

------

#### 3. 作用域属性（控制发送范围）

- **Domain (域名)**：
  - **含义**：指定哪些域名及其子域名可以访问该 Cookie。
  - **面试重点**：设置 `Domain=.google.com` 后，其子域名 `mail.google.com` 也能访问。如果不设置，默认为当前域名（但不包含子域名）。
- **Path (路径)**：
  - **含义**：指定哪些路径下可以访问。
  - **例子**：设置 `Path=/shop`，那么 `/shop/cart` 可以读到，但 `/blog` 读不到。通常为了通用，会设置为 `Path=/`。

------

#### 4. 安全增强属性

这是面试官最想听到的部分，关系到你的安全意识。

**HttpOnly (最重要)**

- **含义**：禁止 JavaScript 通过 `document.cookie` 访问该 Cookie。
- **作用**：**防御 XSS 攻击**。即使攻击者通过脚本注入获取了执行权，也无法偷走用户的 Session ID 或 Token。

**Secure**

- **含义**：该 Cookie 只能在 **HTTPS** 协议下被发送。
- **作用**：防止 Cookie 在不安全的 HTTP 明文传输中被中间人截获（窃听）。

**SameSite**

用来防御 **CSRF（跨站请求伪造）** 攻击，它有三个值：

1. **Strict**：最严：完全禁止跨站发送 Cookie。只有当你从当前域名点开当前页面的链接时，才发 Cookie。
2. **Lax (默认值)**：适中：允许部分第三方请求（如从百度搜到你的网站并点击进入）携带 Cookie，但禁止 POST、iframe 等自动触发的跨站请求携带。
3. **None**：最松：允许跨站发送。但必须配合 `Secure` 属性同时使用。



## 跨域

**同源**是指：**协议、域名、端口** 三者完全一致。只要其中一个不同，就是跨域。

### 跨域的分类：简单请求 vs 预检请求

只要满足以下条件：

- 方法是 `GET`、`POST`、`HEAD`。
- Content-Type 是 `text/plain`、`multipart/form-data`、`application/x-www-form-urlencoded`。

**流程：** 直接发送请求，浏览器在响应头里检查 `Access-Control-Allow-Origin`。



###  预检请求 (Preflight Request)

如果使用了 `PUT`、`DELETE` 或者 `application/json`，浏览器会先发一个 **`OPTIONS`** 请求。

- **目的：** 询问服务器：“我等下要发个 JSON，你允许吗？”
- **服务器响应：** 如果允许，浏览器才发送真正的请求。



### 防止跨域

#### CORS (跨源资源共享) - **最优解**

这是现代开发的正统方案，完全由后端控制。

- **原理：** 服务端在响应头里添加 `Access-Control-Allow-Origin: *`（或指定域名）。
- **重要参数：**
  - `Access-Control-Allow-Methods`: 允许的动词。
  - `Access-Control-Allow-Credentials: true`: 允许携带 Cookie（此时 Origin 不能为 `*`）。

#### Nginx 反向代理 - **工程最常用**

- **原理：** “跨域”只存在于浏览器。服务器之间通信是没有跨域限制的。
- **做法：** 前端请求自己的服务器 `/api`，Nginx 把请求转发给真正的后端。在浏览器看来，你始终在访问同源地址。

#### Node.js 中间件 (Webpack DevServer)

- **场景：** 仅用于**开发环境**。
- **原理：** 启动一个本地代理服务器（Proxy），逻辑同 Nginx。

#### JSONP (过时但常问)

- **原理：** 利用 `<script>` 标签不受同源策略限制的特性。
- **缺点：** **只支持 GET 请求**，且存在安全风险（容易被注入恶意脚本）。、



## HTTP和HTTPS的区别

| **特性**     | **HTTP (HyperText Transfer Protocol)** | **HTTPS (HTTP over SSL/TLS)**                |
| ------------ | -------------------------------------- | -------------------------------------------- |
| **安全性**   | **明文传输**，内容可能被窃听、篡改     | **加密传输**，身份验证，防止中间人攻击       |
| **默认端口** | **80**                                 | **443**                                      |
| **证书**     | 无需证书                               | 需要向 **CA 机构**申请/购买数字证书          |
| **性能**     | 响应速度快，无需握手解密               | 较慢（需 SSL 握手），但 H2/H3 优化后差异极小 |
| **搜索排名** | 无优待                                 | 谷歌等搜索引擎会给予更高的 **SEO 权重**      |

## HTTPS加密的过程

在一切开始之前，服务器（Server）需要先去 **CA（数字证书认证机构）** 申请一份数字证书。

- 证书里包含：**服务器的公钥**、域名信息、有效期、以及 **CA 的电子签名**。
- 这是为了防止“中间人攻击”，确保你访问的网站是真的。

#### 核心阶段：SSL/TLS 握手（以 TLS 1.2 为例）

握手的目标只有一个：**安全地协商出一个用于后续通信的“对称密钥”。**

#### Client Hello（客户端打招呼）

客户端（浏览器）给服务器发个包，包含：

- 支持的 **TLS 版本**（如 1.2 或 1.3）。
- 支持的 **加密算法列表**（密码套件）。
- 一个**随机数 $R_1$ (Client Random)**。

#### Server Hello & Certificate（服务端回应）

服务器收到后回复：

- 确认使用的 **TLS 版本** 和 **加密算法**。
- 发送服务器的 **数字证书**（包含服务器公钥）。
- 又一个**随机数 $R_2$ (Server Random)**。

#### 客户端验证（证书检查）

浏览器收到证书后，会去系统中查找内置的受信任根证书：

- 验证证书是否过期、域名是否匹配。
- 验证 **CA 签名** 是否合法。
- **关键点**：验证通过后，浏览器从证书里取出**服务器的公钥**。

#### Client Key Exchange（生成预主密钥）

浏览器生成第三个**随机数 $R_3$ (Pre-master Secret)**。

- 浏览器用**服务器的公钥**加密 $R_3$，发给服务器。
- **安全性**：由于只有服务器有对应的**私钥**，所以只有服务器能解开 $R_3$。

#### 生成最终的“会话密钥”

现在，客户端和服务器手里都有了三个数：**$R_1 + R_2 + R_3$**。

- 双方各自通过相同的算法，把这三个随机数混合，算出同一个 **会话密钥 (Master Secret)**。
- 后续所有的网页内容（HTTP 数据）都用这个 Master Secret 进行 **对称加密** 传输。

#### 第六步：Finished（握手确认）

双方互发一个加密过的确认包，表示：“以后咱们就用这个密钥聊天了，握手结束！”





## GET请求和POST请求的区别

| **特性**          | **GET**                                 | **POST**                               |
| ----------------- | --------------------------------------- | -------------------------------------- |
| **语义 (最重要)** | **获取**资源（查询）                    | **传输/创建**资源（提交）              |
| **数据位置**      | URL 的 Query String 中                  | 请求体 (Request Body) 中               |
| **数据大小**      | 受 URL 长度限制（约 2KB-8KB）           | 理论上无限制                           |
| **安全性**        | **低**（参数明文暴露在地址栏/历史记录） | **相对高**（不暴露在 URL，但抓包可见） |
| **幂等性**        | **幂等**（多次请求结果一致）            | **非幂等**（多次请求可能创建多条记录） |
| **缓存**          | 可被浏览器主动缓存                      | 默认不缓存                             |
