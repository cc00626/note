### vite和webapck的区别

**构建方式不同**

1. `Webpack` 是一个传统的打包工具，它将应用程序的源代码打包成一个或多个捆绑文件，包括所有的依赖关系。
2. `**Vite**` 则是一个基于现代浏览器原生`ESM`模块支持的构建工具，它不需要打包整个应用程序，而是按需编译和加载模块，减少了构建和重载的时间。

**热更新机制不同**

1. `Webpack`项目中，每次修改文件，都会对整个项目重新进行打包，这对大项目来说，是非常不友好的。虽然`webpack`现在有了缓存机制，但还是无法从根本上解决这个问题。
2. `**Vite**`项目中，监听到文件变更后，会用`websocket`通知浏览器，重新发起新的请求，只对该模块进行重新编译，然后进行替换。

- 并且基于`es module`的特性，`vite`利用浏览器的缓存策略，针对源码模块（我们自己写的代码）做了协商缓存处理，针对依赖模块（第三方库）做了强缓存处理，这样我们项目的访问的速度也就更快了。

**打包方式不同**

- `Vite`在开发环境下不会进行传统的打包，而是将依赖项进行按需编译，并生成原生ES模块的文件。这意味着在生产环境中，`Vite`会生成基于`Rollup`的优化打包结果，利用ES模块的特性进行代码拆分和优化
- `Webpack`会将所有依赖项打包成一个或多个捆绑包，包含了所有需要的资源，如JavaScript、CSS、图片等。

**生态系统不同**

- `Webpack` 有一个庞大的生态系统和丰富的插件支持，可以处理各种场景和需求。
- `Vite` 相对较新，生态系统相对较小，但它仍然受到了很多开发者的欢迎，并且在不断发展壮大。



### vite如何加载commonjs

通过预构建

**回答：**
依赖预构建是 Vite 启动时对第三方依赖进行 **统一打包、转换和缓存**的过程：

- **核心功能**：

1. 将 CommonJS、UMD 或非 ESM 格式的 npm 包转换为浏览器可识别的 ESM。
2. 合并深层依赖，减少浏览器 HTTP 请求数量。
3. 缓存处理结果，避免每次启动都重复编译。

- **解决问题**：

1. **浏览器兼容性**：原生 ESM 不支持 CJS/UMD，预构建解决依赖无法直接加载的问题。
2. **性能优化**：合并深层依赖、减少请求数量，提升冷启动和首次加载速度。
3. **依赖一致性**：处理重复或冲突依赖，保证项目模块图稳定。
4. **HMR 支撑**：预构建后，开发阶段 HMR 只需要更新用户源代码模块，第三方依赖无需重复编译。

机制流程：Vite 扫描入口依赖 → esbuild 转换为 ESM → 生成 `.vite/deps` 缓存 → 浏览器按需加载。

### vite如何兼容老浏览器

Vite 默认只支持现代浏览器，因为它依赖 ES Module。
 如果需要兼容老浏览器，可以使用官方插件 `@vitejs/plugin-legacy`。
 这个插件会在构建时生成两套代码：一套 modern bundle 给支持 ESM 的浏览器，一套 legacy bundle 给旧浏览器，同时通过 Babel 转译语法并自动注入 polyfill。浏览器会根据 `type="module"` 和 `nomodule` 自动选择对应脚本。
